# Settings Page - OTP-Based Account Changes

## Overview
The Settings page allows users to securely update their account information (username, phone number, and password) using OTP verification. This two-step process ensures that only authorized users can make changes to their account.

## Features

### Two-Step Verification Process
1. **Step 1: Settings Selection**
   - Users select which fields they want to change using checkboxes
   - Available fields: Username, Phone Number, Password
   - Form validation ensures at least one field is selected

2. **Step 2: OTP Verification**
   - OTP is sent to the user's registered email
   - User enters the 6-digit OTP
   - Summary box shows which changes will be applied
   - Upon verification, changes are applied to the database

### Security Features
- **OTP Generation**: Backend generates random 6-digit OTP
- **OTP Expiration**: OTP expires after 10 minutes
- **Failed Attempts**: Limits to 3 failed OTP attempts per request
- **Password Hashing**: New passwords are hashed using bcryptjs (10 salt rounds)
- **Email Verification**: OTP sent via Twilio SendGrid for email verification

### User Features
- **Resend OTP**: Users can request a new OTP if they don't receive it
- **Back Button**: Users can go back and modify their selections
- **Field Validation**: Comprehensive validation for all input fields
- **Success/Error Messages**: Clear feedback on all operations

## Frontend Implementation

### Component Structure (`Frontend/src/pages/Settings.jsx`)

#### State Management
```javascript
- currentUsername: Current logged-in username
- step: Current step ('settings' or 'otp-verification')
- error: Error message display
- success: Success message display
- isLoading: Loading state for async operations
- generatedOtp: OTP generated by backend (stored for reference)
- enteredOtp: OTP entered by user
- settingsData: Form data (username, phone, passwords)
- changedFields: Tracks which fields user wants to change
```

#### Key Functions

##### `handleFieldToggle(field)`
Toggles checkbox state for a specific field (username, phone, password)

##### `validateSettings()`
Validates the settings form before sending OTP:
- At least one field must be selected
- Username: Must not be empty
- Phone: Must be at least 10 digits
- Password: Must be 8+ chars, match confirmation, and be different from current

##### `handleSubmit(e)`
Sends OTP request to backend:
- Calls `/api/send-otp-settings` with username and email
- Transitions to OTP verification step on success
- Shows error on failure

##### `handleOtpSubmit(e)`
Verifies OTP and updates settings:
- Calls `/api/update-settings` with OTP and new values
- Backend verifies OTP and applies changes
- Resets form on success

##### `handleResendOtp()`
Requests a new OTP:
- Calls `/api/resend-otp-settings`
- Allows users to get another OTP if they didn't receive first one

#### Form Sections

1. **Current Username Display**: Shows current username (read-only)
2. **Change Username**: Checkbox + input for new username
3. **Change Phone Number**: Checkbox + input for new phone number
4. **Change Password**: Checkbox + 3 inputs (current, new, confirm password)

#### OTP Verification Section
- OTP input field (6 digits only)
- Summary box showing changes to be applied
- Verify button and Back button
- Resend OTP option

### Styling (`Frontend/src/styles/Settings.css`)

#### Key CSS Classes
- `.settings-container`: Main container with white background and shadow
- `.fieldset`: Individual setting sections with hover effects
- `.checkbox-group`: Checkbox and label styling
- `.summary-box`: Highlights what changes will be applied
- `.error-message`: Red error display

- `.otp-input-container`: Centers and styles OTP input

#### Responsive Design

## Backend Implementation

### Routes (`Backend/routes/settings.js`)

#### Endpoints

##### `POST /api/send-otp-settings`
Sends OTP to user's email for settings changes

**Request Body:**
```json
{
  "username": "johndoe",
  "email": "user@example.com"
}
```

**Response:**
```json
{
  "success": true,
  "message": "OTP sent to your email",
  "testOTP": "123456"  // For testing only - remove in production
}
```

**Features:**
- Generates random 6-digit OTP
- Stores OTP with 10-minute expiration
- Sends OTP via Twilio SendGrid
- Returns `testOTP` for development/testing

##### `POST /api/update-settings`
Verifies OTP and updates user settings

**Request Body:**
```json
{
  "username": "johndoe",
  "email": "user@example.com",
  "otp": "123456",
  "newUsername": "newjohndoe",
  "newPhone": "+1-555-0101",
  "newPassword": "newSecurePassword123",
  "changedFields": {
    "username": true,
    "phone": false,
    "password": true
  }
}
```

**Response:**
```json
{
  "success": true,
  "message": "Settings updated successfully",
  "data": {
    "username": "newjohndoe",
    "email": "user@example.com",
    "phone": "+1-555-0101",
    "passwordUpdated": true,
    "usernameUpdated": true,
    "phoneUpdated": false
  }
}
```

**Validation:**
- Verifies OTP is correct (limits to 3 attempts)
- Checks OTP hasn't expired (10 minutes)
- Verifies username matches OTP request
- Hashes password if changing (bcryptjs, 10 rounds)

##### `POST /api/verify-otp-settings`
Verifies OTP without updating settings (optional pre-verification)

**Request Body:**
```json
{
  "email": "user@example.com",
  "otp": "123456"
}
```

**Response:**
```json
{
  "success": true,
  "message": "OTP verified successfully"
}
```

##### `POST /api/resend-otp-settings`
Resends OTP to user's email

**Request Body:**
```json
{
  "email": "user@example.com",
  "username": "johndoe"
}
```

**Response:**
```json
{
  "success": true,
  "message": "OTP resent to your email",
  "testOTP": "654321"  // For testing only
}
```

**Features:**
- Prevents spam by checking if OTP still valid
- Returns 429 Too Many Requests if OTP recently sent
- Generates new OTP and stores with fresh expiration

### OTP Storage
Currently uses in-memory store (`otpStore` object):
```javascript
otpStore[email] = {
  otp: "123456",
  expiresAt: 1234567890,
  attempts: 0,
  username: "johndoe"
}
```

**Note**: For production, replace with Redis or database (Mongoose, Sequelize, etc.)

### Database Integration
The current implementation uses mock data. To integrate with a database:

1. **Replace OTP storage**: Use Redis for fast OTP lookups with automatic expiration
2. **User updates**: Replace mock user data with database queries
3. **Update endpoint**: Query database with Mongoose/Sequelize to update user records

Example structure for database:

## Testing

## Security Considerations

1. **Never log OTP to user**: Log only for debugging, remove in production
2. **HTTPS Required**: Always use HTTPS in production
3. **Email Verification**: Ensures only email owner receives changes
4. **Password Hashing**: All new passwords hashed with bcryptjs
5. **Rate Limiting**: Consider adding rate limiting to OTP endpoints
6. **CSRF Protection**: Add CSRF tokens in production
7. **Session Management**: Verify user is authenticated before allowing changes

## Future Enhancements

1. **Email Verification**: Confirm email changes before applying
2. **Two-Factor Authentication**: Add backup codes or authenticator app
3. **Change History**: Log all account changes with timestamps
4. **IP Verification**: Add warning if change from new IP
5. **Backup Verification Methods**: SMS or authenticator app as backup
6. **Activity Log**: Show recent activities and logins

## Troubleshooting

### OTP Not Received
- Check spam folder
- Verify email address is correct
- Click "Resend OTP"
- Check Twilio SendGrid configuration

### "OTP has expired" Error
- OTP expires after 10 minutes
- Click "Resend OTP" to get a new one

### Settings Not Updating
- Verify OTP is correct
- Check database connection (if using database)
- Check browser console for errors

### CORS Errors
- Verify CORS is enabled in backend
- Check API URL is correct
- Ensure credentials are included if needed
